# 邮件知识库处理系统 - 完整使用流程

## 🔄 系统概述

本系统用于将电子邮件(.eml文件)批量处理并上传到GPTBots知识库，支持内网部署和多用户协作。

---

## 📱 **阶段0：访问系统**

### 用户操作
```
浏览器输入: http://10.52.20.100:3000
```

### 系统处理
1. 前端从部署电脑加载（Next.js应用）
2. 自动检测hostname = `10.52.20.100`
3. 设置 `API_BASE_URL = http://10.52.20.100:5001`
4. 所有后续API请求都发到这个地址

---

## 📤 **阶段1：批次管理 - 上传邮件**

### 用户操作步骤
1. 点击 "批次管理" → "新增批次"
2. 输入批次标签（如"2024年10月客户咨询"）
3. 选择邮件文件（.eml格式）
4. 点击上传

### 系统处理逻辑

#### 前端（用户浏览器）

**1️⃣ 选择文件后 → 调用 API: `/api/check-duplicates`**
- 发送: 文件名列表
- 返回: 哪些是重复的

**2️⃣ UI显示**
- 重复文件: 黄色背景 + "跳过"标签
- 统计: "检测到 X 个已处理过的邮件"

**3️⃣ 点击上传 → POST `/api/upload`**
- 文件通过HTTP上传到 `10.52.20.100:5001`

#### 后端（部署电脑）

**4️⃣ 接收文件**
- 生成批次ID: `batch_20241024_153045_x2k9`
- 创建目录: `eml_process/uploads/batch_xxx/`

**5️⃣ 重复检测**
- 遍历所有批次目录
- 收集已存在的文件名
- 跳过重复文件（不保存）

**6️⃣ 保存文件**
```
message-1-001.eml → 保存 ✅
message-1-002.eml → 跳过（重复）⚠️
message-1-003.eml → 保存 ✅
```

**7️⃣ 创建元数据** (`.batch_info.json`)
```json
{
  "batch_id": "batch_20241024_153045_x2k9",
  "custom_label": "2024年10月客户咨询",
  "file_count": 2,
  "status": {
    "uploaded": true,
    "cleaned": false,
    "llm_processed": false,
    "uploaded_to_kb": false
  }
}
```

### 结果
- ✅ 新邮件保存在部署电脑
- ⚠️ 重复邮件被跳过
- 📊 批次卡片显示在列表中

## ⚙️ **阶段2：全自动处理 - 配置**

### 用户操作步骤
1. 点击 "全自动运行"
2. 查看批次列表，勾选要处理的批次
3. （可选）展开配置选择特定的LLM/知识库
4. 选择目标知识库（必选）
5. 点击 "开始全自动处理"

### 系统处理逻辑

**1️⃣ 加载环境配置**
```
GET /api/config/env
├─ 后端读取 .env 文件
├─ 返回 3个LLM Keys, 3个KB Keys
└─ 前端显示在选择框中
```

**2️⃣ 加载批次列表**
```
GET /api/batches
├─ 显示批次卡片
├─ 高亮显示处理状态
└─ 默认隐藏已完成批次 ✅
```

**3️⃣ 获取知识库列表**
```
POST /api/knowledge-bases
├─ 使用选中的KB API Key
├─ 调用内网API: 10.52.20.41:19080
└─ 返回可用知识库列表
```

**4️⃣ 计算预计时间**
```
统计选中批次的文件总数
估算: 4.5秒/文件
显示预计用时
```

## 🤖 **阶段3：自动处理 - 执行**

系统自动执行3个步骤，用户只需等待即可。

### 步骤1: 邮件清洗去重 (Clean)

**API调用**
```
POST /api/auto/clean
参数:
  - batch_ids: ["batch_xxx", "batch_yyy"]
  - skip_if_exists: true
```

**处理流程**

**1️⃣ 智能跳过检查**
```
检查 eml_process/processed/batch_xxx/
如果已有.md文件 → 跳过该批次
记录: "批次已有X个处理后的文件，跳过清洗"
```

**2️⃣ 解析EML文件**
```
读取: eml_process/uploads/batch_xxx/*.eml
解析邮件头:
  - 发件人 (From)
  - 收件人 (To)
  - 主题 (Subject)
  - 时间 (Date)
提取正文: 纯文本/HTML → 转文本
生成结构化数据
```

**3️⃣ 批次内去重**
```
计算内容Hash (MD5)
相同Hash → 标记为重复
只保留唯一邮件
```

**4️⃣ 生成Markdown**
```markdown
## [邮件主题]

**发件人**: xxx@example.com
**收件人**: yyy@example.com
**时间**: 2024-10-24 15:30:45

---

[清洗后的邮件正文]
```

**5️⃣ 保存结果**
```
保存到: eml_process/processed/batch_xxx/
  ├─ message-1-001.md ✅
  ├─ message-1-002.md (被去重跳过)
  └─ message-1-003.md ✅

更新批次状态: cleaned = true
```

**⏱️ 耗时**: ~0.5秒/文件

---

### 步骤2: LLM内容处理 (LLM Process)

**API调用**
```
POST /api/auto/llm-process
参数:
  - batch_ids: ["batch_xxx"]
  - api_key: "app-JfbLb7yTAgtelQLZu7PgWBOE"
  - skip_if_exists: true
```

**处理流程**

**1️⃣ 智能跳过检查**
```
检查 eml_process/final_output/batch_xxx/
如果已有.md → 跳过
```

**2️⃣ 读取已清洗文件**
```
eml_process/processed/batch_xxx/*.md
```

**3️⃣ LLM优化处理**（逐个文件）
```
调用内网API:
POST http://10.52.20.41:19080/v2/conversation/message
{
  "bot_id": "xxx",
  "user_id": "auto_processor",
  "query": "[邮件内容]"
}

LLM处理:
  ├─ 内容总结
  ├─ 关键信息提取
  ├─ 格式优化
  └─ 语义增强

返回优化后的Markdown
```

**4️⃣ 保存结果**
```
保存到: eml_process/final_output/batch_xxx/
  └─ message-1-001.md (LLM优化版)
```

**5️⃣ 更新状态**
```
llm_processed = true
```

**⏱️ 耗时**: ~2-3秒/文件（API调用）

### 步骤3: 上传到知识库 (Upload to KB)

**API调用**
```
POST /api/auto/upload-kb
参数:
  - batch_ids: ["batch_xxx"]
  - kb_api_key: "app-qJ3OvJto9sh2QWn9HnoijhD7"

  - chunk_mode: "token"

```

**处理流程**

**1️⃣ 智能跳过检查**
```
读取批次元数据
如果 uploaded_to_kb = true → 跳过
```

**2️⃣ 读取LLM处理后的文件**
```
eml_process/final_output/batch_xxx/*.md
```

**3️⃣ 上传到知识库**（逐个文件）
```
调用内网API:
POST http://10.52.20.41:19080/v1/knowledge_base/upload
{
  "knowledge_base_id": "68b94099...",
  "file_name": "message-1-001.md",
  "content": "[Markdown内容]",
  "chunk_token": 600
}

知识库处理:
  ├─ 按600 token分块
  ├─ 向量化
  └─ 存储到知识库
```

**4️⃣ 自动获取知识库名称**
```
GET 知识库列表
匹配 knowledge_base_id
保存到批次元数据: kb_name = "Email"
```

**5️⃣ 更新状态**
```

```

**⏱️ 耗时**: ~1秒/文件

## 🔄 **智能特性**

### 1. 增量处理
```
第一次处理批次A → 完整处理（清洗→LLM→上传）
第二次处理批次A → 自动跳过（已完成）
第三次处理批次A和B → 只处理B
```

### 2. 三层去重机制

**第一层：上传阶段**
- 检测：文件名在其他批次中是否存在
- 处理：UI提示，不保存重复文件
- 目的：节省存储空间

**第二层：清洗阶段**
- 检测：批次内邮件内容是否重复（MD5 Hash）
- 处理：只保留唯一邮件
- 目的：避免批次内重复

**第三层：跨批次检查**
- 检测：全局记录已处理的邮件
- 处理：跳过已在其他批次处理过的文件
- 目的：避免重复处理和上传

### 3. 状态管理
```
每个批次独立状态跟踪
可以中断后继续处理
不会丢失已处理的数据
支持批量选择处理
```

### 4. 预计时间算法
```
平均处理时间 = 4.5秒/文件

组成:
  - 清洗去重: 0.5秒
  - LLM处理: 2-3秒（API调用）
  - 知识库上传: 1秒

计算公式:
  预计时间 = 文件总数 × 4.5秒
  
示例:
  10个文件 → 45秒
  26个文件 → 117秒（约2分钟）
  100个文件 → 450秒（约7.5分钟）
```

## ⚠️ **注意事项**

### 文件格式要求
```
✅ 支持: .eml 格式（标准电子邮件格式）
❌ 不支持: .msg, .pst, .txt, .pdf
```

### 批次标签建议
```
✅ 好的标签:
  - "2024-10-客户咨询-产品A"
  - "销售部-Q3季度邮件"
  - "技术支持-紧急问题"
  
❌ 不好的标签:
  - "邮件"（太笼统）
  - "123"（无意义）
  - 留空（必填项）
```

### 处理时间预估
```
文件数量 | 预计时间 | 说明
---------|----------|------
1-10     | 1分钟内  | 适合快速测试
11-50    | 2-4分钟  | 日常使用
51-100   | 5-8分钟  | 批量处理
100+     | 8分钟+   | 建议分批处理
```

### 系统限制
```
单批次文件数: 建议 ≤ 100个
并发批次数: 建议 ≤ 5个
文件大小: 建议 ≤ 10MB/个
知识库容量: 根据GPTBots配额
```

## 🔧 **故障排除**

### 问题1: 无法访问系统
```
检查:
  1. 部署电脑是否开机
  2. 服务是否启动（3000和5001端口）
  3. 防火墙是否允许
  4. IP地址是否正确
```

### 问题2: 配置无法加载
```
检查:
  1. 部署电脑上的.env文件是否存在
  2. .env文件格式是否正确
  3. API Keys是否有效
  4. 查看后端日志: logs/api_server.log
```

### 问题3: 处理失败
```
检查:
  1. GPTBots服务是否正常 (10.52.20.41:19080)
  2. API Key是否有效
  3. 知识库ID是否正确
  4. 查看日志: logs/activity.log
```

### 问题4: 上传重复文件
```
这是正常的！
  - 系统会自动检测并标记重复文件
  - 重复文件不会被保存
  - 不会重复上传到知识库
```
